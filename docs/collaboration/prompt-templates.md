# Prompt 模板

本页根据 `CODEX_PROMPT_TEMPLATE.md` 重组，没有逐字照搬原文，而是按团队协作场景重新分类。

## 1. 默认开发

适合常规功能开发、修复 bug 和迭代修改。

使用时应明确：

- 目标项目路径或项目名
- 需求内容
- 是否怀疑远端已经有变更
- 修改后需要验证哪些页面或接口

模板重点：

- 先读规则和 `kooboo.d.ts`
- 先看邻近同类实现
- 做最小必要修改
- 跑本地 verify
- 最后给出远端验证建议

## 2. 快速修复

适合目标清晰、改动很局部的场景。

模板重点：

- 只改必要文件
- 保持现有目录约定
- 修改后说明是否可以 `kb push`

## 3. 代码审查

适合在 push 前检查风险。

建议重点检查：

- 是否违反目录语义规则
- 是否和当前 `kooboo.d.ts` 冲突
- 是否误写成普通 Node / Express / Koa 风格
- 是否存在只有 push 后才暴露的 runtime 风险

## 4. 验证状态

适合只判断“当前是否适合 push”。

模板重点：

- 读取规则和 `kooboo.d.ts`
- 跑 verify
- 输出远端验证 checklist

## 5. 先 Pull 判断

适合你不确定远端是否已经发生变化的场景。

模板重点：

- 判断是否应该先 `kb pull`
- 给出原因
- 顺手指出最可能涉及的目录与文件

## 6. 远程验证

适合本地改动完成后，生成明确的远端检查清单。

建议明确：

- 打开哪些页面
- 请求哪些接口
- 预期结果是什么
- 失败时优先检查哪些文件

## 代码示例

```text
请检查 <项目路径> 当前状态是否适合执行 kb push。
要求：
- 先读取规则和 kooboo.d.ts
- 运行已有 verify
- 如果通过，输出远端验证 checklist
```
